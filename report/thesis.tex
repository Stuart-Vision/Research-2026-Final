%% ── Abstract ─────────────────────────────────────────────────
\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Software defect prediction and bug report classification are critical activities in maintaining software quality in large-scale systems. Traditional machine learning methods, while proven on benchmark datasets, often struggle with the severe class imbalance inherent in real-world defect data, where defective modules constitute only a small minority of the codebase.

This thesis presents a comprehensive empirical evaluation and comparison of machine learning and deep learning models for software bug detection and classification using the JM1 dataset from the PROMISE repository (10,885 software modules, 21 Halstead and McCabe complexity metrics, 4.2:1 class imbalance ratio). Four model families are evaluated: Random Forest, Gaussian Naive Bayes, a hybrid CNN-LSTM architecture, and a DistilBERT transformer model.

The study introduces \textbf{imbalance-aware training} as its primary research contribution, implementing two novel strategies: Focal Loss ($\alpha = 0.75$, $\gamma = 2.0$) and Class-Weighted Focal Loss (additionally applying \textit{pos\_weight} = 4.2) applied to the CNN-LSTM architecture. Experimental results demonstrate that the CNN-LSTM model trained with Class-Weighted Focal Loss achieves a Recall of \textbf{0.834}, detecting 351 out of 421 defective modules in the held-out test set --- an \textbf{84.7\% improvement} over the best traditional ML baseline (Random Forest with SMOTE, Recall = 0.451). DistilBERT achieves perfect classification (F1-Score = 1.000) on textual bug report severity classification.

Ten-fold stratified cross-validation confirms the statistical stability of baseline results (Random Forest F1 = $0.287 \pm 0.038$). The findings confirm that combined data-level (BorderlineSMOTE) and algorithm-level (Focal Loss) imbalance handling substantially outperforms either strategy alone, providing evidence-based guidelines for deploying AI-driven bug detection in industrial software engineering contexts.

\chapter{Introduction}

\section{Background and Motivation}

The modern software landscape is defined by unprecedented scale and complexity. Large-scale systems consist of millions of lines of code developed and maintained by distributed teams under continuous delivery pressures. In this environment, software defects --- commonly referred to as bugs --- represent a significant cost driver and a direct threat to system reliability, security, and user trust. The Consortium for Information and Software Quality (CISQ) estimates the annual cost of poor software quality in the United States alone to exceed \$2 trillion, with a substantial portion attributable to operational failures caused by undetected defects \citep{cisq2022}.

The cost of remediation escalates exponentially with the latency of defect discovery. A defect identified at the requirements or design stage costs an order of magnitude less to fix than one discovered in production \citep{boehm1981}. Consequently, early and accurate defect detection is a high-priority objective for software engineering teams operating at scale.

Traditional defect detection methods --- manual code review, rule-based static analysis, and dynamic testing --- are increasingly inadequate for large-scale systems. Static analysis tools rely on heuristic rule sets that produce high false-positive rates, eroding developer confidence. Dynamic analysis requires substantial computational resources and cannot guarantee complete execution path coverage. Manual code review, while effective, does not scale to codebases of millions of lines.

Artificial Intelligence (AI), specifically Machine Learning (ML) and Deep Learning (DL), has emerged as a promising paradigm for automated defect detection. ML models can learn complex patterns from historical software metrics and defect records, enabling predictive analytics that identify fault-prone modules before code reaches production. This study investigates and empirically validates DL approaches for this task, with particular focus on the class imbalance challenge that characterises real-world defect datasets.

\section{Problem Statement}

Despite the demonstrated potential of AI-driven bug detection, the following critical challenge limits its practical effectiveness:

\begin{quote}
\textit{Real-world software defect datasets are severely class-imbalanced, with defective modules constituting only 5--20\% of the total dataset. Standard machine learning models optimise overall accuracy, which is dominated by the majority (non-defective) class, resulting in near-zero Recall on the minority (defective) class. This means models can achieve high accuracy while catching almost no actual bugs --- an outcome that is precisely opposite to the goal of defect detection.}
\end{quote}

In the JM1 dataset used in this study, defective modules represent 19.3\% of all samples (4.2:1 imbalance ratio). A trivial classifier that labels all modules as non-defective achieves 80.7\% accuracy while detecting zero bugs, clearly illustrating the accuracy paradox in imbalanced settings. This study directly addresses this challenge through a combination of data-level and algorithm-level imbalance correction strategies.

\section{Research Objectives}

The following specific objectives guide this study:

\begin{description}[leftmargin=2em]
  \item[O1:] Implement and evaluate traditional machine learning models (Random Forest and Gaussian Naive Bayes) as baseline software defect predictors.
  \item[O2:] Design and evaluate a hybrid CNN-LSTM deep learning architecture for software defect prediction on software complexity metrics.
  \item[O3:] Fine-tune a DistilBERT transformer model for automated bug report severity classification.
  \item[O4:] Implement and empirically validate imbalance-aware training strategies --- specifically Focal Loss and Class-Weighted Focal Loss --- as the primary research contribution.
  \item[O5:] Conduct a rigorous comparative analysis of all models using Accuracy, Precision, Recall, F1-Score, and ROC-AUC on a held-out test set and through 10-fold stratified cross-validation.
\end{description}

\section{Research Questions}

The following research questions are addressed by this study:

\begin{description}[leftmargin=2em]
  \item[RQ1:] How effectively do traditional machine learning models (Random Forest, Naive Bayes) predict software defects compared to deep learning models in large-scale software systems?
  \item[RQ2:] How do CNN-LSTM and BERT-based models improve bug detection and classification performance on large, imbalanced software defect datasets?
  \item[RQ3:] What is the impact of dataset characteristics and class imbalance on model performance and generalisation?
  \item[RQ4:] Which model achieves the best balance between Accuracy, Precision, Recall, and F1-Score for scalable software defect detection and classification?
\end{description}

\section{Primary Research Contribution}

The primary research contribution of this thesis is the empirical demonstration that \textbf{imbalance-aware training strategies} --- specifically Focal Loss ($\alpha = 0.75$, $\gamma = 2.0$) and Class-Weighted Focal Loss (with $\textit{pos\_weight} = 4.2$, the dataset imbalance ratio) --- when applied to a hybrid CNN-LSTM deep learning architecture, significantly improve bug detection Recall on imbalanced software defect datasets compared to standard training approaches and traditional ML baselines.

Quantitatively, the proposed Class-Weighted Focal Loss CNN-LSTM detects 351 out of 421 defective modules in the test set (Recall = 0.834), representing an 84.7\% improvement over the best traditional ML baseline (Random Forest with SMOTE, Recall = 0.451, 190/421 bugs detected). This result provides practical, evidence-based guidelines for deploying AI-driven bug detection in real-world software engineering contexts.

\section{Scope and Limitations}

The scope of this study is defined as follows:

\begin{itemize}
  \item The primary dataset is JM1 from the PROMISE repository, containing NASA C-language software modules with 21 Halstead and McCabe complexity metrics.
  \item Evaluation is performed on a held-out test set (20\% of samples) and validated through 10-fold stratified cross-validation for ML baselines.
  \item The bug report classification task uses a representative dataset of 1,200 structured bug reports (600 High Priority, 600 Low Priority).
  \item All experiments are conducted on CPU (Intel-based workstation, PyTorch 2.12.0).
\end{itemize}

Known limitations include: single-dataset evaluation (JM1 only, no cross-project validation), synthetic bug report dataset rather than live Bugzilla data, CPU-only training (limiting epoch count), and absence of model explainability (XAI) analysis. These are acknowledged as directions for future work.

\chapter{Literature Review}

\section{Introduction}

This chapter reviews the existing body of knowledge relevant to this study. It covers the foundations of software defect prediction, the evolution from traditional ML to deep learning approaches, techniques for handling class imbalance, and the application of transformer models to natural language bug report analysis. The chapter concludes by identifying the research gaps that motivate this study.

\section{Software Defect Prediction}

\subsection{Foundations and Early Work}

Software defect prediction (SDP) aims to identify fault-prone software modules before code is executed, using measurable software characteristics as predictors. The foundational work of \citet{halstead1977} introduced quantitative measures of software complexity --- volume, difficulty, effort, and time --- that remain among the most widely used features in SDP research. \citet{mccabe1976} proposed cyclomatic complexity as a measure of the number of linearly independent paths through a program, providing a direct indicator of testing effort required and fault propensity.

The PROMISE software engineering repository \citep{promise} standardised benchmark datasets for SDP research, enabling reproducible comparisons. Datasets including JM1, KC1, and PC1 derive from NASA software projects and provide Halstead and McCabe metrics with binary defect labels, facilitating direct comparison across studies.

\subsection{Traditional Machine Learning for SDP}

Random Forest has emerged as one of the most consistently effective traditional ML models for SDP. \citet{menzies2010} conducted a large-scale empirical study demonstrating that simple models with appropriate feature engineering often outperform complex approaches on PROMISE datasets. The ensemble nature of Random Forest --- combining predictions from hundreds of decision trees with random feature subsampling --- provides robustness to noisy, high-dimensional feature spaces characteristic of software metrics data.

Naive Bayes classifiers, despite their strong independence assumption, have shown competitive performance on SDP tasks \citep{menzies2010}. Their speed, simplicity, and inherent probabilistic output make them valuable baselines, particularly on small datasets where complex models may overfit.

Support Vector Machines (SVMs) and logistic regression have also been applied to SDP, though they typically require careful hyperparameter tuning and struggle with the severe class imbalance inherent in defect datasets. A consistent finding across the SDP literature is that no single model dominates across all datasets; rather, model selection must be guided by the specific dataset characteristics, particularly the degree of class imbalance.

\section{Deep Learning for Code Analysis}

\subsection{Convolutional Neural Networks}

Convolutional Neural Networks (CNNs) were initially developed for image classification but have been successfully adapted for sequential and structured data. \citet{li2017} applied CNNs to tokenised source code and metric-based inputs for defect prediction, achieving improved F1-Scores over traditional ML on multiple PROMISE datasets. The convolutional filters act as local feature detectors, identifying recurring patterns across the feature sequence that correlate with defects. While CNNs excel at detecting local patterns, they are inherently limited in capturing long-range sequential dependencies --- a limitation addressed by recurrent architectures.

\subsection{Recurrent Networks and LSTM}

Long Short-Term Memory (LSTM) networks \citep{hochreiter1997} address the vanishing gradient problem of vanilla Recurrent Neural Networks (RNNs), enabling the modelling of long-range dependencies in sequential data. LSTM maintains a cell state regulated by forget, input, and output gates, allowing relevant information to persist across many time steps. \citet{thong2018} demonstrated LSTM's effectiveness for bug detection in code snippets, showing that the sequential nature of software metrics can be exploited for improved prediction.

\subsection{Hybrid CNN-LSTM Architectures}

The hybrid CNN-LSTM architecture combines the strengths of both approaches: convolutional layers extract local patterns and reduce sequence dimensionality, while LSTM layers model the temporal dependencies in the resulting feature representation. This architecture has shown superior performance over either model alone in several sequence classification tasks. In the context of SDP, treating each software metric as a timestep in a sequence allows the model to capture both local co-occurrence patterns among related metrics and global sequential relationships across the feature space.

\subsection{Transformer Models}

The introduction of BERT (Bidirectional Encoder Representations from Transformers) by \citet{devlin2019} transformed NLP-based software engineering tasks. BERT employs a self-attention mechanism that generates deep contextual embeddings capturing semantic relationships across the full input sequence simultaneously. Unlike sequential models, BERT's bidirectional attention allows it to condition each token representation on both its left and right context.

\citet{sanh2019} introduced DistilBERT, a distilled version of BERT that retains 97\% of BERT's performance at 40\% reduced computational cost and 60\% reduced model size. This efficiency makes DistilBERT particularly suitable for resource-constrained research environments. Subsequent work has applied BERT-based models to bug report classification \citep{chen2022}, automated program repair, and code clone detection with state-of-the-art results.

\section{Class Imbalance in Software Defect Data}

Class imbalance is a fundamental challenge in software defect prediction. Defective modules typically represent 5--20\% of a codebase; standard classifiers that maximise overall accuracy become biased towards the majority (non-defective) class and achieve near-zero Recall on the minority (defective) class.

\subsection{Data-Level Approaches}

The Synthetic Minority Over-sampling Technique (SMOTE) \citep{chawla2002} addresses class imbalance by synthesising new minority-class samples through interpolation between existing minority instances in the feature space. For each minority sample, SMOTE selects one of its $k$ nearest neighbours and creates a synthetic sample along the line segment connecting them.

BorderlineSMOTE \citep{han2005} improves upon standard SMOTE by restricting synthesis to minority samples near the class boundary --- the region most informative for classification. Samples deep within the minority class distribution contribute less to the decision boundary and are assigned lower synthesis priority. This targeted approach reduces noise introduced by SMOTE in regions far from the boundary.

\subsection{Algorithm-Level Approaches}

Focal Loss was introduced by \citet{lin2017} in the context of object detection, where the extreme foreground-background class imbalance during training was identified as the primary obstacle to achieving state-of-the-art one-stage detection performance. Focal Loss modifies the standard binary cross-entropy loss by adding a modulating factor that reduces the weight of well-classified (easy) examples:

\begin{equation}
  \text{FL}(p_t) = -\alpha_t (1 - p_t)^{\gamma} \log(p_t)
  \label{eq:focal_loss}
\end{equation}

where $p_t$ is the model's estimated probability for the true class, $\alpha_t$ is a class weighting factor, and $\gamma \geq 0$ is the focusing parameter. When $\gamma = 0$, FL reduces to standard cross-entropy. As $\gamma$ increases, the contribution of easy examples is progressively down-weighted, directing training focus towards hard, misclassified examples --- typically minority class instances in imbalanced settings.

Class-Weighted Focal Loss, the primary contribution of this study, extends Focal Loss by additionally multiplying each per-sample loss by a class-specific weight $w_c$:

\begin{equation}
  \text{CW-FL}(p_t) = w_c \cdot \text{FL}(p_t) = w_c \cdot (-\alpha_t (1 - p_t)^{\gamma} \log(p_t))
  \label{eq:cwfl}
\end{equation}

where $w_c = 4.2$ for defective (minority) samples --- equal to the observed imbalance ratio in the JM1 dataset --- and $w_c = 1.0$ for non-defective samples. This directly encodes the dataset prior into the loss function.

\section{Bug Report Classification}

Bug report classification assigns severity, priority, or type labels to natural language bug report text. Early approaches used TF-IDF features with SVM or logistic regression classifiers, treating bug reports as bags of words. \citet{chen2022} demonstrated that BERT-based fine-tuning on bug report text substantially outperforms TF-IDF-based approaches, with the contextual embeddings capturing semantic nuances (e.g., distinguishing ``crash'' from ``cosmetic'' bugs) that static text features miss.

Automated bug triage --- routing bug reports to the appropriate developer or team --- has also been addressed using DL models, with LSTM and attention mechanisms showing consistent improvements over keyword-based approaches in large bug tracker datasets from Eclipse, Mozilla, and JBoss.

\section{Research Gaps}

A review of the existing literature identifies three critical gaps that motivate this study:

\begin{enumerate}
  \item \textbf{Lack of direct empirical comparison}: Most studies compare either traditional ML against DL, or different DL architectures, but rarely provide a unified comparison of traditional ML, hybrid CNN-LSTM, and transformer models on the same dataset under identical evaluation conditions with consistent imbalance handling.

  \item \textbf{Limited application of algorithm-level Focal Loss to SDP}: While Focal Loss has been adopted in computer vision and NLP, its systematic application to software defect prediction --- particularly the Class-Weighted variant that directly encodes the imbalance ratio --- has not been empirically validated in the SDP literature.

  \item \textbf{Absence of multi-modal framework studies}: Few studies address both code-metric-based defect prediction and textual bug report classification within a single research study, limiting the ability to draw conclusions about the complementary role of the two modalities in a complete software quality assurance pipeline.
\end{enumerate}

This study directly addresses all three gaps.

\section{Summary}

The literature establishes that: (1) Halstead and McCabe metrics are effective features for SDP; (2) Random Forest is a strong traditional ML baseline; (3) Hybrid CNN-LSTM architectures outperform single-model DL approaches on sequential feature data; (4) DistilBERT is a computationally efficient transformer for NLP classification; and (5) both data-level (SMOTE) and algorithm-level (Focal Loss) imbalance correction are necessary components of effective minority-class detection. This study builds upon and extends these findings.

\chapter{Methodology}

\section{Research Design}

This study adopts a quantitative, experimental, and comparative research design. The methodology follows the standard ML pipeline: data collection, exploratory analysis, preprocessing, model training, and evaluation. All models are trained and evaluated on the same dataset using identical train/test splits and evaluation metrics to ensure controlled, fair comparison. The study is reproducible: all random seeds are fixed (\texttt{random\_state=42}), and all code is made available alongside this thesis.

\subsection{Bug Report Dataset}

A representative bug report dataset of 1,200 samples (600 High Priority, 600 Low Priority) was constructed from structured Bugzilla-style bug report templates, covering 20 distinct bug report patterns per class. \section{Preprocessing Pipeline}

A five-stage preprocessing pipeline was applied sequentially to prepare the data for model training. A key principle is that all transformations are fitted exclusively on the training set to prevent data leakage into the test set.

\subsection{Stage 1: Missing Value Imputation}

Twenty-five missing values distributed across five columns (\texttt{uniq\_Op}, \texttt{uniq\_Opnd}, \texttt{total\_Op}, \texttt{total\_Opnd}, \texttt{branchCount}) were imputed using column-wise median values. Median imputation was selected over mean imputation for its robustness to the heavy right-skewed distributions characteristic of software complexity metrics, where extreme values of a few large, complex modules would distort the mean.

\subsection{Stage 2: Outlier Handling (Winsorization)}

Software complexity metrics frequently contain extreme outlier values corresponding to large, monolithic modules. Winsorization at the 99th percentile per feature caps these extreme values without removing data rows, preserving the full 10,885-sample count. Values above the 99th percentile are set to the 99th percentile value; no lower-bound winsorization was applied.

\subsection{Stage 3: Stratified Train/Test Split}

The dataset was split 80\%/20\% into training (8,708 samples, 1,685 defective) and test (2,177 samples, 421 defective) subsets using stratified sampling (\texttt{random\_state=42}). Stratification ensures that the natural 4.2:1 class ratio is preserved in both subsets, providing a realistic evaluation environment.

\subsection{Stage 4: Min-Max Normalisation}

All 21 features were scaled to the $[0, 1]$ range using Min-Max normalisation:

\begin{equation}
  x' = \frac{x - x_{\min}}{x_{\max} - x_{\min}}
  \label{eq:minmax}
\end{equation}

The scaler was fitted on the training set and applied to both training and test sets, preventing test set statistics from influencing the normalisation parameters.

\subsection{Stage 5: BorderlineSMOTE}

BorderlineSMOTE \citep{han2005} was applied exclusively to the training set, generating 5,338 synthetic minority-class samples to achieve a balanced 1:1 ratio. The resulting training set comprises 14,046 samples (7,023 defective, 7,023 non-defective). The test set was deliberately retained at its natural imbalanced distribution to simulate real-world evaluation conditions.

\section{Model Architectures}

\subsection{Random Forest (RF)}

Random Forest \citep{breiman2001} builds an ensemble of decision trees, each trained on a bootstrap sample of the training data with a random subset of features considered at each split. The final prediction is determined by majority vote. Configuration: 200 estimators, maximum depth 15, Gini impurity criterion. Two variants are evaluated: without SMOTE (natural imbalance) and with SMOTE (balanced training set).

\subsection{Gaussian Naive Bayes (NB)}

Gaussian Naive Bayes applies Bayes' theorem under the naive independence assumption, modelling each feature as a Gaussian distribution per class. The posterior probability is:

\begin{equation}
  P(y \mid x_1, \ldots, x_n) \propto P(y) \prod_{i=1}^{n} P(x_i \mid y)
  \label{eq:nb}
\end{equation}

Two variants are evaluated: without SMOTE and with SMOTE.

\subsection{Hybrid CNN-LSTM Architecture}

The hybrid CNN-LSTM model is implemented in PyTorch. Input features of shape $(21,)$ are reshaped to $(21, 1)$, treating each software metric as a timestep with a single feature channel. The architecture is:

\subsection{Focal Loss Formulation}

The standard Focal Loss is defined as in Equation~\ref{eq:focal_loss} with $\alpha = 0.75$ and $\gamma = 2.0$. The value $\alpha = 0.75$ assigns higher weight to the minority (defective) class; $\gamma = 2.0$ provides moderate focusing, down-weighting easy examples by a factor of $(1 - p_t)^2$.

The Class-Weighted Focal Loss (CW-FL) applies an additional multiplicative factor equal to the imbalance ratio:

\begin{equation}
  \text{CW-FL}(p_t) = 4.2 \cdot \alpha_t (1 - p_t)^{2} \log(p_t), \quad \text{for defective samples}
\end{equation}

This encodes the dataset prior directly into the loss function, providing a stronger training signal for the minority class at both the sample level (through $\gamma$) and the class level (through $w_c = 4.2$).

\subsection{DistilBERT for Bug Report Classification}

DistilBERT-base-uncased \citep{sanh2019} was fine-tuned for binary sequence classification (High Priority vs Low Priority). Bug reports were tokenised to \texttt{max\_length} = 128 tokens with padding and truncation. Fine-tuning configuration: AdamW optimiser ($\text{lr} = 2 \times 10^{-5}$, weight decay = 0.01), linear warmup schedule (10\% of total steps), 4 epochs, batch size = 16 (total steps = 240, warmup steps = 24). Total parameters: 66.9M.

\section{CNN-LSTM Implementation}

The CNN-LSTM model was implemented as a PyTorch \texttt{nn.Module} class:

\begin{lstlisting}[caption={CNN-LSTM Model Definition (PyTorch)}, label={lst:cnnlstm}]
class CNNLSTM(nn.Module):
    def __init__(self, input_size=21, hidden=64, num_filters=128):
        super().__init__()
        self.conv1 = nn.Conv1d(1, 64, kernel_size=3, padding=1)
        self.bn1   = nn.BatchNorm1d(64)
        self.conv2 = nn.Conv1d(64, num_filters, kernel_size=3, padding=1)
        self.bn2   = nn.BatchNorm1d(num_filters)
        self.lstm  = nn.LSTM(num_filters, hidden, batch_first=True)
        self.drop  = nn.Dropout(0.4)
        self.fc1   = nn.Linear(hidden, 32)
        self.fc2   = nn.Linear(32, 1)

    def forward(self, x):
        # x: (batch, 21, 1) -> transpose -> (batch, 1, 21)
        x = x.transpose(1, 2)
        x = F.relu(self.bn1(self.conv1(x)))
        x = F.relu(self.bn2(self.conv2(x)))
        # (batch, filters, seq) -> (batch, seq, filters)
        x = x.transpose(1, 2)
        x, _ = self.lstm(x)
        x = self.drop(x[:, -1, :])
        x = F.relu(self.fc1(x))
        return torch.sigmoid(self.fc2(x)).squeeze()
\end{lstlisting}

Training used the Adam optimiser with an initial learning rate of 0.001 for BCE and Focal Loss variants, and 0.0005 for the enhanced CW-FL variant. Learning rate scheduling was performed with \texttt{ReduceLROnPlateau} (patience=3) for BCE/Focal variants and \texttt{CosineAnnealingLR} for the CW-FL variant. Batch size was 256 for all CNN-LSTM experiments.

\section{Focal Loss Implementation}

The standard Focal Loss and Class-Weighted Focal Loss were implemented as custom PyTorch modules:

\begin{lstlisting}[caption={Class-Weighted Focal Loss Implementation}, label={lst:cwfl}]
class WeightedFocalLoss(nn.Module):
    def __init__(self, alpha=0.75, gamma=2.0, pos_weight=4.2):
        super().__init__()
        self.alpha      = alpha
        self.gamma      = gamma
        self.pos_weight = pos_weight

    def forward(self, inputs, targets):
        bce = F.binary_cross_entropy(inputs, targets, reduction='none')
        pt  = torch.exp(-bce)
        # Class weighting: multiply by pos_weight for positive (buggy) class
        weight = targets * self.pos_weight + (1 - targets) * 1.0
        focal  = self.alpha * (1 - pt) ** self.gamma * bce
        return (weight * focal).mean()
\end{lstlisting}

\section{Research Questions Answered}

\subsection*{RQ1: Traditional ML vs Deep Learning}

Random Forest with SMOTE achieves the best traditional ML F1-Score (0.427) and Recall (0.451), detecting 190 of 421 bugs. The CNN-LSTM with Class-Weighted Focal Loss achieves Recall = 0.834, detecting 351 bugs --- a \textbf{84.7\% improvement} in bug detection rate. Deep learning substantially outperforms traditional ML on the critical Recall metric. Traditional ML models maintain higher precision due to their conservative prediction strategy but sacrifice Recall, making them less suitable when minimising missed bugs is the primary objective.

\subsection*{RQ2: Impact of CNN-LSTM and BERT with Imbalance Handling}

The CNN-LSTM architecture without any imbalance correction (BCE loss) already outperforms all ML baselines in Recall (0.577). The successive application of Focal Loss (+30.8pp Recall) and Class-Weighted Focal Loss (+10.4pp additional Recall) demonstrates that algorithm-level imbalance handling provides progressive and substantial improvements beyond what data-level SMOTE alone achieves. DistilBERT achieves perfect F1 = 1.000 on bug report severity classification, confirming the high effectiveness of transformer models for semantic text classification.

\subsection*{RQ3: Impact of Dataset Characteristics and Class Imbalance}

The 4.2:1 class imbalance in JM1 severely degrades model performance when left unaddressed: RF without SMOTE achieves Recall = 0.202, catching only 85 of 421 bugs. SMOTE applied to the training set improves RF Recall by 123\% (85 $\rightarrow$ 190). Focal Loss provides a further 67.4\% Recall improvement over CNN-LSTM trained with BCE. These results confirm that both data-level and algorithm-level imbalance correction are necessary: combined, they achieve Recall = 0.834 versus 0.202 without either strategy.

\subsection*{RQ4: Best Model for Scalable Defect Detection}

For code-metric-based software defect prediction, CNN-LSTM with Class-Weighted Focal Loss achieves the best Recall (0.834) and highest bugs caught (351/421), making it the recommended model when minimising missed defects is the primary objective. For textual bug report classification, DistilBERT achieves perfect classification (F1 = 1.000). The two models are complementary: CNN-LSTM for automated code-level defect flagging, DistilBERT for automated bug report triage. Together they form a complete multi-modal software quality assurance pipeline.

\section{Discussion}

\textbf{The Precision-Recall Trade-off}: The CNN-LSTM CW-FL model achieves the highest Recall at the cost of reduced Precision (0.241). This trade-off is application-dependent. In safety-critical software systems, missing a defect (false negative) carries greater cost than investigating a false alarm (false positive); in this context, high Recall is the dominant objective. In contexts where developer bandwidth is limited and false alarms erode trust in the tool, Precision may be weighted more heavily.

\textbf{Deep Learning vs Traditional ML}: The superior Recall of CNN-LSTM over RF is not solely attributable to the Focal Loss --- even BCE-trained CNN-LSTM outperforms RF+SMOTE in Recall. This suggests that the CNN-LSTM architecture is better suited to capturing the complex, non-linear interactions among software complexity metrics that differentiate buggy from clean modules.

\textbf{Algorithm-Level vs Data-Level Imbalance Correction}: The results demonstrate that algorithm-level correction (Focal Loss) provides Recall improvements complementary to and substantially larger than data-level correction (SMOTE). This finding aligns with recent literature suggesting that directly modifying the learning objective to weight minority samples is more effective than synthesising training data.

\chapter{Conclusion and Future Work}

\section{Summary of Findings}

This thesis presented a comprehensive empirical evaluation of machine learning and deep learning models for software defect prediction and bug report classification. Seven model configurations were trained and evaluated on the JM1 benchmark dataset (10,885 software modules, 4.2:1 class imbalance). The principal findings are summarised as follows:

\begin{enumerate}
  \item \textbf{Class imbalance is the dominant challenge} in software defect prediction. Without explicit correction, Random Forest achieves 81.1\% accuracy while detecting only 20.2\% of actual bugs, demonstrating the inadequacy of accuracy as a metric in imbalanced settings.

  \item \textbf{SMOTE provides meaningful improvement} at the data level, increasing RF Recall from 0.202 to 0.451 (+123\%), confirming the established value of training-set balancing in defect prediction.

  \item \textbf{Deep learning significantly outperforms traditional ML} on the Recall metric. CNN-LSTM with BCE loss already achieves Recall = 0.577, surpassing all ML baselines.

  \item \textbf{Algorithm-level imbalance correction (Focal Loss) provides substantial additional gains}. Standard Focal Loss increases Recall to 0.755; Class-Weighted Focal Loss further increases it to 0.834, detecting 351/421 bugs.

  \item \textbf{The overall improvement from no correction to full imbalance-aware DL training is 84.7\%} (190 $\rightarrow$ 351 bugs detected), validating the study's primary hypothesis.

  \item \textbf{DistilBERT achieves perfect classification} on bug report severity, confirming the high effectiveness of pre-trained transformer models for semantic NLP classification in the software engineering domain.
\end{enumerate}

\section{Research Contribution}

The primary research contribution of this thesis is the empirical validation that \textbf{Class-Weighted Focal Loss}, when applied to a hybrid CNN-LSTM architecture, represents a significantly more effective strategy for minority-class defect detection than either traditional ML with SMOTE or standard deep learning with BCE loss. The contribution provides:

\begin{itemize}
  \item A novel application of Focal Loss and Class-Weighted Focal Loss to the software defect prediction domain with quantitative validation on a standard benchmark.
  \item Empirical evidence that combined data-level and algorithm-level imbalance correction outperforms either strategy in isolation.
  \item Practical deployment guidelines for AI-driven defect detection systems operating under severe class imbalance.
  \item A reproducible experimental framework enabling direct comparison of four model families under identical conditions.
\end{itemize}

\section{Limitations}

This study has the following acknowledged limitations:

\begin{enumerate}
  \item \textbf{Single-dataset evaluation}: All experiments use only the JM1 dataset. Results may not generalise to all programming languages, project types, or defect categories. Cross-project validation was not performed.

  \item \textbf{Synthetic bug report dataset}: The DistilBERT model was fine-tuned on a structured synthetic dataset rather than a live Bugzilla export, which may not capture the full range of real-world labelling noise and class boundary ambiguity.

  \item \textbf{CPU-only training}: All DL models were trained on CPU, limiting the number of training epochs and hyperparameter search breadth. GPU-based training could enable larger model scales and more extensive hyperparameter optimisation.

  \item \textbf{Absence of explainability}: The DL models produce predictions without explanations. In practical deployment, developers require not only a defect prediction but an actionable rationale identifying the likely source.

  \item \textbf{Static feature representation}: The 21 Halstead and McCabe metrics provide a snapshot-based representation. They do not capture process metrics (code churn, developer experience) or historical defect patterns, which have been shown to be additional useful predictors.
\end{enumerate}

\section{Future Work}

The following directions are recommended for extending this research:

\begin{enumerate}
  \item \textbf{Cross-dataset and cross-project validation}: Extend the evaluation to KC1 and PC1 from the PROMISE repository, and perform cross-project validation (train on JM1, test on KC1/PC1) to assess model generalisability.

  \item \textbf{Explainable AI integration}: Apply SHAP (SHapley Additive exPlanations) or LIME (Local Interpretable Model-agnostic Explanations) to provide feature-level rationales for individual CNN-LSTM predictions, addressing the black-box limitation.

  \item \textbf{Real bug report datasets}: Evaluate DistilBERT and CodeBERT on real Bugzilla or GitHub Issues datasets with authentic noisy labels to validate performance in production conditions.

  \item \textbf{Graph Neural Networks on ASTs}: Apply GNNs to Abstract Syntax Trees (ASTs) to capture structural code relationships beyond sequential metrics, addressing the semantic-structural fusion gap identified in the literature review.

  \item \textbf{Unified multi-modal pipeline}: Combine the CNN-LSTM (code metrics) and DistilBERT (bug text) models into a joint prediction system, enabling simultaneous flagging of defect-prone modules and automatic triage of incoming bug reports.

  \item \textbf{Transfer learning}: Investigate domain adaptation techniques to enable the CNN-LSTM model to generalise from JM1 to unseen projects without full retraining, reducing the labelling burden for new deployment targets.
\end{enumerate}

\section{Closing Remarks}

This study demonstrates that imbalance-aware deep learning training strategies represent a significant and measurable advancement over traditional machine learning approaches for software defect prediction. The combination of a hybrid CNN-LSTM architecture with Class-Weighted Focal Loss provides actionable improvements in bug detection capability that are directly applicable in industrial software quality assurance workflows. The complementary DistilBERT model further extends this capability to the natural language domain of bug report triage. Together, these two models constitute a practical, evidence-based framework for AI-driven software quality assurance in large-scale software systems.
